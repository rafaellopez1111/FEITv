O projeto FEITv consiste no desenvolvimento de um sistema de gerenciamento de vídeos, criado com o objetivo de oferecer ao usuário uma plataforma completa de interação com conteúdos audiovisuais, permitindo a busca, organização e manipulação de vídeos de forma simples e estruturada. A aplicação foi desenvolvida utilizando a linguagem Java e estruturada com base no padrão arquitetural MVC (Model-View-Controller), possibilitando uma separação clara entre as responsabilidades do sistema e garantindo maior organização, reutilização de código e facilidade de manutenção. A interface gráfica foi construída utilizando Java Swing por meio da utilização de diversas telas (JFrames), enquanto a persistência de dados é realizada através de um banco de dados PostgreSQL, acessado por meio da camada DAO.

O funcionamento do sistema inicia-se a partir da classe principal, responsável por abrir a interface inicial e permitir que o usuário realize o cadastro ou o login. Após essa etapa, o controle do fluxo passa para o Controller responsável pela autenticação, que valida as credenciais e identifica o tipo de usuário. Esse processo está concentrado no Controller\_Login, que além de validar os dados inseridos, também armazena o ID do usuário logado utilizando a classe Storage\_User, permitindo que as demais funcionalidades do sistema possam operar com base no contexto do usuário atual. A partir dessa validação, o sistema redireciona automaticamente o usuário para a interface correspondente, seja ela a área de usuário comum ou a área administrativa. 

Uma vez autenticado, o fluxo do sistema passa a ser controlado pelo Controller_Inicial, que atua como um ponto central de navegação. Esse controller é responsável por métodos como paginabuscar, paginalistar, paginacurtidas e paginaplaylist, todos utilizados para trocar de telas de forma dinâmica com base na interação do usuário. Cada um desses métodos cria uma nova instância de um JFrame específico, inicializa seu controller correspondente e torna a nova tela visível, ao mesmo tempo em que oculta a tela anterior. Esse comportamento permite uma navegação fluida entre as diferentes funcionalidades do sistema, mantendo a lógica de controle centralizada.

As funcionalidades relacionadas aos vídeos estão encapsuladas principalmente no Controller\_Videos, que trabalha diretamente com a camada DAO para realizar operações sobre os dados de vídeos. Esse controller implementa métodos responsáveis por buscar vídeos a partir do nome informado pelo usuário, registrar interações como curtidas, descurtidas e dislikes, além de listar vídeos disponíveis no sistema. No caso do administrador, o mesmo controller também contém funções que permitem criar novos vídeos e excluir conteúdos existentes, utilizando os dados inseridos nas interfaces específicas de cadastro. A interação com o banco é feita através da VideoDAO, que executa comandos SQL utilizando uma conexão previamente estabelecida, garantindo que os dados sejam persistidos corretamente. 

O gerenciamento de playlists e vídeos favoritos é realizado pelo Controller\_Playlist, que possui funções específicas para criação, edição e exclusão de playlists, além de operações para adicionar e remover vídeos da lista de favoritos do usuário. Um ponto relevante dessa parte do sistema é o uso de uma constante interna chamada de playlist de favoritos, que padroniza a identificação da lista principal do usuário. Esse controller também utiliza o Storage\_User para associar ações ao usuário correto e interage diretamente com PlaylistDAO e VideoDAO para manipulação dos dados. Dessa forma, quando o usuário adiciona um vídeo aos favoritos, o controller executa uma operação que envolve tanto a identificação do usuário quanto a associação do vídeo na tabela correspondente.

Além do gerenciamento direto de playlists, existe o Controller\_inicial\_favoritos, que atua como um controlador de navegação dentro da área de favoritos. Ele é responsável por direcionar o usuário para telas específicas relacionadas ao gerenciamento de playlists e de vídeos favoritos, funcionando como um intermediário entre a interface principal e as funcionalidades detalhadas desse módulo. De maneira semelhante, o Controller\_inicial\_admin organiza a navegação da interface administrativa, centralizando o acesso às funções exclusivas de gestão do sistema. 
Outra parte importante do sistema está presente no controller principal genérico (Controller), que concentra operações relacionadas à manipulação de usuários. Esse controller é responsável por funções como cadastro de novos usuários, consulta de dados existentes e listagem de informações no sistema, sendo utilizado em conjunto com telas específicas como a de consulta de usuários e a de visualização de estatísticas. A partir dessas funções, o administrador pode acessar dados relevantes do sistema e acompanhar seu funcionamento de forma mais ampla. 

Toda a comunicação com o banco de dados é realizada através da camada DAO, que funciona como uma ponte entre os controllers e o PostgreSQL. Cada DAO possui métodos próprios para executar operações específicas, utilizando objetos de conexão fornecidos pela classe Conexao. Essa classe é responsável por estabelecer a comunicação com o banco por meio do uso de DriverManager, utilizando parâmetros como endereço, porta e credenciais armazenadas em variáveis de ambiente. Esse modelo permite maior segurança e flexibilidade, além de manter o código organizado ao separar completamente a lógica de persistência do restante do sistema.

A interface gráfica desempenha um papel fundamental na experiência do usuário, sendo composta por diversas telas implementadas com JFrame. Cada interface possui botões e campos de entrada que disparam eventos, os quais são tratados diretamente pelos controllers. Por exemplo, na tela de busca de vídeos, ao clicar no botão de busca, o método correspondente no Controller\_Videos é acionado para consultar o banco e retornar os resultados. Da mesma forma, ações como clicar em “Curtir”, “Adicionar aos favoritos” ou “Criar playlist” são convertidas em chamadas diretas aos métodos dos controllers, que processam a ação e atualizam os dados.

O fluxo geral do sistema segue um padrão consistente, no qual cada ação realizada na interface percorre um caminho bem definido: a View captura o evento, o Controller interpreta a ação, o DAO executa a operação no banco de dados e, por fim, o resultado é refletido novamente na interface. Essa integração entre as camadas garante que o sistema funcione de forma organizada, evitando acoplamento excessivo e permitindo que cada parte seja desenvolvida e mantida de forma independente.
